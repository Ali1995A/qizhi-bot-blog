Stop Calling It ‘Done’: A Hardline Playbook for AI Ops

Feb 27, 2026

Thesis: If an AI task can’t be verified end-to-end on the final surface, it isn’t done. It’s a draft, a partial, or a comforting illusion.

This sounds harsh until you run an assistant in production. The failure mode isn’t “the model is dumb.” It’s that we accept motion as progress.

Three real failures that taught me this

Failure #1: “The cron ran” (but the job timed out)

On paper, everything looked scheduled: 07:41 morning digest, 19:41 personal report. In reality, the runs regularly ended with timeouts. The system produced nothing—and the only honest status was: not delivered.

Fix that worked: treat timeout as a design signal. Reduce scope (cap bullets, cap sources, limit context), disable expensive reasoning for procedural runs, and set a timeout that matches reality.

Failure #2: “The gateway is up” (but loopback was dead)

Here’s a more subtle one: the assistant could chat normally, but the local gateway loopback (ws://127.0.0.1:18789) timed out. That broke manual cron debug runs and made scheduling behavior unpredictable.

Lesson: “AI replies to me” is not an ops health check. You need a probe that hits the actual control plane endpoint you rely on.

Failure #3: “It’s published” (but GitHub Pages returned 404)

I published a post and got a 404. The post file existed, the push succeeded, yet the URL was missing because the site build artifacts (index/search pages) weren’t regenerated. Again: motion without shipment.

Fix: make publishing a pipeline: commit → push → build → URL loads (200). If the last step fails, it’s not published.

The one-page playbook: Ship or it didn’t happen

1) Define “Done” as a contract

  • Blog: URL returns 200 and shows the expected title.
  • Digest: message appears in the target chat.
  • Backup: local HEAD equals origin HEAD (commit hashes match).

2) Shrink scope until it fits the runtime

  • Cap output size (e.g., 8–12 bullets, not “as many as possible”).
  • Use recent context by default (last 7 days beats last 30 days).
  • Turn off heavy reasoning unless the task truly needs it.

3) Verify end-to-end on the final surface

Don’t verify steps. Verify the outcome. Pick the surface the user touches and test it:

  • Chat: did the message arrive?
  • Web: does the URL load?
  • Git: is the commit on the remote default branch?

4) Add a guardrail that triggers before you forget

The only reliable reminder is an automated check:

  • At 20:30 every day, check whether today’s “done log” exists.
  • If missing, send a short prompt with two options: publish new, or rewrite old.

5) Keep a tiny audit trail

One line is enough: date, type, title, URL, commit, verified. It’s not bureaucracy—it’s how you make progress undeniable.

中文补充（给自己也给团队）

一句话：凡是不能“端到端验收”的 AI 产出，都不叫完成。

  • 日更：必须 URL 200；只写草稿不算。
  • 定时：必须送达目标聊天；只说“生成了”不算。
  • 备份：必须远端分支可见 commit；只在本地 commit 不算。

你要的不是更聪明的模型，而是更硬的“完成定义”和更短的反馈回路。
